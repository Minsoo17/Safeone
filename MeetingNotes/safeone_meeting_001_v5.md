## 1. 프로젝트 개요

### 1.1 서비스명
- **세이프원 (SafeOne)** - 가칭

### 1.2 핵심 정의
> 안전업체와 안전요원(구직자)을 연결하는 매칭 플랫폼

### 1.3 시장 구조 이해

**현재 계약 흐름:**
```
학교 → (학교장터/나라장터) → 안전업체 → (네이버 카페) → 안전요원
```

**핵심 인사이트:**
- 학교는 플랫폼에 직접 접근하지 않음 (공공조달 시스템 이용)
- 세이프원의 타겟은 **안전업체 ↔ 안전요원** 구간

### 1.4 안전요원 배치 기준 (교육부 지침)

| 규모 | 학생 수 | 안전요원 배치 기준 |
|------|--------|-------------------|
| 소규모 | 100명 미만 | 1명 이상 의무 |
| 중규모 | 100~149명 | 1명 이상 의무 |
| 대규모 | 150명 이상 | 50명당 1명 의무 (버스 1대당 1인 권장) |
| 국외 | 전체 | 50명당 1명 의무 (규모 무관) |

## 2. 벤치마킹: (주)디스탑안전교육

### 2.1 기업 개요
| 항목 | 내용 |
|------|------|
| 대표 | 전선희 (대한민국 수학여행안전요원 1호) |
| 설립 | 2014년 (세월호 참사 이후) |
| 실적 | 2,000여 건 안전활동, 120여 명 안전요원 보유 |
| 연락처 | 1666-1724 |

### 2.2 현재 구인 방식
1. 네이버 카페에 구인글 게시
2. 선착순 댓글 접수
3. 개별 연락 → 자격 검증 → 계약서 작성

### 2.3 현재 방식의 문제점
- 선착순 방식으로 인력 품질 검증 어려움
- 매칭 과정이 수동적이고 시간 소요
- 이력/평판 관리 시스템 부재
- 노쇼 및 부적격 인력 리스크

## 3. 수익 모델 설계

### 3.1 기본 원칙
> "안전요원은 무료, 안전업체에서 수익 창출"

**논리적 근거:**
- 안전요원 유료화 시 가입 욕구 저하
- 플랫폼 핵심 자산 = 인력풀 (양면시장의 보조금 측)
- 안전업체의 기존 업무(카페 관리, 개별 연락, 계약서 작성)를 대체하면 비용 지불 의향 발생

### 3.2 단계별 수익 모델

```
[1단계] 안전요원 무료 가입 → 인력풀 확보 (임계치: 100명 이상)
            ↓
[2단계] 안전업체 유료화 → 핵심 매출원
            ↓
[3단계] 안전요원 프리미엄 → 부가 매출 (자기 홍보 목적 수수료)
            ↓
[4단계] 보험사 제휴 → 수수료 + 우회 방지 + 신뢰도
            ↓
[5단계] 데이터 사업 → 리스크 분석, 보험 상품 개발 협업
```

### 3.3 구독료 기반 수익 모델

| 등급 | 월 구독료 | 매칭 건수 | 초과 시 |
|------|----------|----------|--------|
| 브론즈 | 5만원 | 월 10건 | 업그레이드 유도 |
| 실버 | 10만원 | 월 30건 | 업그레이드 유도 |
| 골드 | 15만원 | 무제한 | - |

**건당 수수료 없음** - 구독료에 포함

### 3.4 신규 업체 유입 전략

| 항목 | 내용 |
|------|------|
| 무료 체험 | 첫 1개월 무료 |
| 체험 등급 | **실버 기능** 제공 |
| 체험 종료 후 | 유료 전환 안내 → 등급 선택 |

### 3.5 매칭 우회(탈플랫폼) 방지 전략

| 전략 | 방식 | 효과 |
|------|------|------|
| 정산 플랫폼化 | 계약서, 급여 정산을 플랫폼에서만 | 우회 시 행정 불편 |
| 평가 기록 귀속 | 플랫폼 통해야만 평가 누적 | 안전요원 이용 유인 |
| 구독료 중심 | 건당이 아닌 구독료 중심 | 우회해도 이미 비용 지불 |
| 보험/보증 연계 | 플랫폼 매칭 건만 보험 적용 | 업체 리스크 관리 |

## 5. 매칭 시스템 설계

### 5.1 매칭 방식

| 항목 | 결정 사항 |
|------|----------|
| MVP 방식 | 공고형 (업체가 공고 → 요원이 지원) |
| 이후 추가 | 역제안형 (요원이 가능일 등록 → 업체가 제안) |
| 공고 구조 | 1개 공고 = N명 모집 |

### 5.2 공고 등록 항목

| 항목 | 필수 | 예시 |
|------|------|------|
| 행사명 | ✅ | "○○초등학교 제주 수학여행" |
| 행사 일시 | ✅ | 2025.04.15 ~ 04.17 (2박 3일) |
| 집합 장소 | ✅ | 서울 ○○초등학교 정문 앞 |
| 목적지 | ✅ | 제주도 |
| 학생 수 | ✅ | 180명 |
| 모집 인원 | ✅ | 4명 |
| 일당 | ✅ | 15만원 (고정 금액 필수 입력) |
| 숙식 제공 여부 | ✅ | 제공 / 미제공 |
| 우대 조건 | ⬜ | "제주 경험자 우대" |
| 상세 안내 | ⬜ | 일정표, 주의사항 등 |

### 5.3 지원자 상태값

| 상태 | 설명 | 요원에게 보이는 문구 |
|------|------|---------------------|
| `지원중` | 검토 대기 | "검토 중입니다" |
| `선발됨` | 매칭 확정 | "선발되었습니다" |
| `대기` | 충원 완료, 추가 선발 가능성 | "대기 중입니다" |
| `선발마감` | 최종 마감 (미선발/거절 통합) | "선발이 완료되었습니다" |
| `취소됨` | 본인 취소 승인됨 | "취소되었습니다" |

### 5.4 중복 지원 정책

| 항목 | 결정 사항 |
|------|----------|
| 중복 지원 | **제한** (같은 날짜에 1개만 지원 가능) |
| 겹침 기준 | **하루라도 겹치면 중복** |
| 체크 대상 | 지원중, 선발됨, 대기 상태만 (선발마감, 취소됨 제외) |

```
[중복 지원 시나리오]
요원 A: 서울 건 지원 완료 (12월 15일 ~ 17일)
    ↓
요원 A: 부산 건 지원 시도 (12월 17일 ~ 18일)
    ↓
시스템: "12월 17일에 이미 지원한 공고가 있습니다" → 지원 불가
```

### 5.5 선발 및 대기자 관리

| 항목 | 결정 사항 |
|------|----------|
| 지원 순서 | 업체에게만 표시 (요원에게는 비공개) |
| 선발 방식 | 업체가 자유롭게 선택 (순서 무관) |
| 대기자 유지 | 충원 완료 후에도 유지 (취소 시 추가 선발용) |
| 대기자 선택 | 업체가 자유롭게 선택 + 지원 순서 참고 가능 |

### 5.6 매칭 종료 조건

| 조건 | 처리 |
|------|------|
| 업체가 "매칭종료" 버튼 클릭 | 즉시 종료, 대기자 전원 "선발 마감" |
| 모집공고 기간 종료 | 자동 종료 |
| 행사 D-day 도래 | 자동 종료 (안전장치) |

### 5.7 미충원 시 처리

| 상황 | 처리 |
|------|------|
| 모집 인원 미달 + D-3 이내 | "긴급 구인" 표시 + 상위 노출 |

## 7. 운영 정책

### 7.1 취소 정책

**취소 시점 기준: 전날 자정(00:00)**

| 취소 시점 | 가능 여부 | 페널티 |
|----------|----------|--------|
| 행사 7일 전 ~ | ✅ 자유 취소 | 없음 (취소 횟수만 기록) |
| 행사 3일 전 ~ 7일 전 | ✅ 취소 가능 | 취소 횟수 기록 |
| 행사 1일 전 ~ 3일 전 | ✅ 취소 가능 | 취소 횟수 + 경고 1회 |
| 행사 1일 전 이내 | ❌ 취소 불가 | 긴급 상황만 관리자 처리 |
| 당일 연락 없음 | 노쇼 | 경고 + 30일 지원 제한 |

**지원 취소 (선발 전)**: 자유롭게 가능, 페널티 없음

### 7.2 취소 사유 관리

| 사유 유형 | 예시 | 페널티 |
|----------|------|--------|
| 단순 변심 | "일정이 안 맞아서", "다른 일감 생겨서" | 누적 카운트 |
| 불가피 사유 | 질병, 가족 경조사, 사고 등 | 증빙 시 면제/감경 |
| 노쇼 | 연락 없이 불참 | 강력 페널티 |

### 7.3 선발 후 취소 플로우

```
[요원] 취소 요청 + 사유 입력
    ↓
[업체] 승인 / 거부
    ↓
승인 시 → 매칭 해제 + 요원에게 "취소 횟수" 기록
거부 시 → 매칭 유지 (요원과 업체가 직접 소통)
```

### 7.4 긴급 취소 플로우 (행사 1일 전 이내)

```
[행사 1일 전 이내]
    ↓
[요원] 긴급 상황 발생 (사고, 입원 등)
    ↓
[시스템] 취소 버튼 비활성화 → "긴급 연락" 버튼만 활성화
    ↓
[요원] 긴급 연락 버튼 클릭 → 사유 입력
    ↓
[관리자] 즉시 알림 수신 (모니터링)
    ↓
[업체] 알림 수신 → 대기자 중 추가 선발 또는 "긴급 구인" 재공고
    ↓
[관리자] 사유 검토 후 페널티 면제/부과 결정
```

### 7.5 노쇼 정책

| 횟수 | 페널티 |
|------|--------|
| 1회 | 경고 + 30일 지원 제한 |
| 2회 | 60일 지원 제한 |
| 3회 이상 | 영구 제한 |

### 7.6 업체 귀책 취소

| 항목 | 결정 사항 |
|------|----------|
| MVP | 보상 없음 (플랫폼 관여 X) |
| 이후 | 취소 수수료 검토 |

### 7.7 분쟁 처리

| 단계 | 설명 |
|------|------|
| 1단계 | 당사자 간 직접 해결 (채팅/연락) |
| 2단계 | 플랫폼에 분쟁 신고 (신고 버튼) |
| 3단계 | 관리자 중재 (증거 검토) |
| 4단계 | 최종 결정 (평가 삭제, 정산 조정 등) |

## 8. 정산/결제 설계

### 8.1 정산 플로우

```
[매칭 확정]
    ↓
[행사 진행]
    ↓
[행사 종료]
    ↓
[업체] 플랫폼에 일당 입금 (3일 이내)
    ↓
[플랫폼] 입금 확인

[플랫폼] 요원에게 지급 (입금 후 3일 이내)
    ├─ 분쟁 없음 → 정상 지급
    └─ 분쟁 신고 → 보류 후 해결 시 지급
```

### 8.2 정산 규칙

| 항목 | 결정 사항 |
|------|----------|
| 정산 방식 | 플랫폼 중개 (업체 → 플랫폼 → 요원) |
| 결제 타이밍 | 후결제 (행사 완료 후) |
| 업체 입금 기한 | 행사 종료 후 **3일 이내** |
| 요원 지급 시점 | 입금 확인 후 **3일 이내** (분쟁 시 보류) |
| 수수료 부담 | 업체 (구독료로 납부) |
| 건당 수수료 | 없음 (구독료에 포함) |
| 일당 표시 | 업체가 **고정 금액 필수 입력** |
| 미입금 시 | 독촉 알림 → 연체 시 이용 제한 |

### 8.3 일당 입력 UI

```
[공고 등록 시]
일당: [      ] 원 (필수 입력, 숫자만)

※ 입력한 금액이 안전요원에게 지급됩니다.
※ 플랫폼 수수료는 별도로 업체에 청구됩니다. (구독료에 포함)
```

## 9. 가입 및 인증 설계

### 9.1 업체 인증 - 완전 자동화 ✅

**국세청 사업자등록정보 API 활용**

```
[업체 가입]
사업자등록번호 + 대표자명 + 개업일 입력
    ↓
[시스템] 국세청 API 자동 호출
    ↓
✅ 일치 + 사업중 → 즉시 승인 → 1개월 무료 체험(실버)
❌ 불일치 → "정보가 일치하지 않습니다" 안내
❌ 폐업/휴업 → "휴폐업 사업자입니다" 안내
```

**API 정보:**
- 공공데이터포털: 국세청_사업자등록정보 진위확인 및 상태조회 서비스
- 1회 100건, 1일 100만건 호출 가능

### 9.2 안전요원 인증 - 자가 인증 + 사후 검증

**대한적십자사 수료증은 공공 API 없음 → 자가 인증 방식**

```
[안전요원 가입]
    ↓
수료증 이미지 업로드 + 수료번호 + 수료일 직접 입력
    ↓
[시스템] 형식 체크 (수료번호 자릿수, 수료일 유효성)
    ↓
✅ 형식 통과 → "임시 승인" (즉시 활동 가능)
    ↓
[사후 검증]
- 첫 매칭 시 업체가 수료증 실물 확인
- 허위 신고 접수 시 관리자 검토
- 허위 판명 시 → 영구 제한 + 기존 평가 삭제
```

## 10. 전체 사용자 시나리오

### 10.1 안전요원 여정

```
[1. 가입]
회원가입 (이메일/소셜) → 본인인증 (휴대폰)
    ↓
기본정보 입력 (이름, 연락처, 지역)
    ↓
자격증 등록 (이미지 + 수료번호 + 수료일 입력)
    ↓

[2. 일감 찾기]
공고 목록 조회 (지역, 날짜, 일당 필터)
    ↓
공고 상세 확인 → 지원하기 (중복 날짜 체크)
    ↓
"검토 중" 상태

[3. 매칭]
업체가 수락 → "선발됨" 알림
    또는
업체가 "매칭종료" 클릭 → 즉시 "선발 마감"
    또는
모집공고 기간 종료 → 자동 "선발 마감"
    또는
충원 완료 but 매칭종료 전 → "대기" 상태

[4. 행사 수행]
집합 → 행사 진행 → 종료

[5. 정산 & 평가]
업체가 플랫폼에 입금 (3일 이내)
    ↓
플랫폼에서 지급 (입금 후 3일 이내, 분쟁 시 보류)
    ↓
업체 평가하기 (14일 이내)
    ↓
평가 공개 (14일 후 일괄)

[6. 재이용]
평가 기록 누적 → 신뢰도 상승 → 다음 일감 지원
```

- 포지셔닝: "단순 매칭 플랫폼" → "안전요원 생태계 인프라"

## 16. 기술 스택 및 개발 환경

### 16.1 개발 영역 정의

| 영역 | 설명 |
|------|------|

| 외부 API 연동 | 국세청 API, 본인인증 API, PG사 결제 |
| 인프라/DevOps | 클라우드 서버, 배포, 모니터링 |
| 알림 시스템 | 푸시 알림, 이메일, SMS |

| **배포** | Vercel (프런트) + AWS EC2/Railway (백엔드) | 쉬운 배포, 확장성 |

## 19. 핵심 결정 사항 요약

| 결정 사항 | 내용 |
|-----------|------|
| 플랫폼 포지션 | 안전업체 ↔ 안전요원 매칭 (학교 직접 접근 X) |
| 초기 타겟 | 수학여행 안전요원 (린 스타트업 접근) |
| 수익 주체 | 안전업체 (안전요원은 무료) |
| 수익 모델 | 구독료 (브론즈 5만/실버 10만/골드 15만) |
| 건당 수수료 | 없음 (구독료에 포함) |
| 신규 업체 | 첫 1개월 무료 (실버 기능) |
| 매칭 방식 | 공고형 (1개 공고 = N명 모집) |
| 중복 지원 | 날짜 겹치면 불가 |
| 매칭 종료 | 업체 수동 + 기간 만료 시 자동 |
| 핵심 경쟁력 | 평가 기록 기반 검증된 인력풀 |
| 평가 항목 | 양방향 5개 항목, 14일 후 일괄 공개 |
| 취소 정책 | 전날 자정 이후 취소 불가, 노쇼 시 30일 제한 |
| 정산 방식 | 플랫폼 중개, 후결제, 업체 3일 내 입금 → 요원 3일 내 지급 |
| 업체 인증 | 국세청 API 자동 검증 |
| 요원 인증 | 자가 인증 + 사후 검증 |
| 긴급 대체 | 업체 자율 (대기자 선발 또는 긴급 구인) |
| 우회 방지 | 정산 플랫폼化 + 평가 귀속 + 보험 연계 |
| 초기 전략 | 디스탑안전교육과 파트너십 (친척 연결고리 활용) |
| 확장 방향 | 보험사 제휴 → 데이터 사업 |
| **기술 스택** | Next.js + NestJS + PostgreSQL + Prisma |
| **개발 도구** | VS Code + GitHub |
| **DB 호스팅** | Neon (PostgreSQL 클라우드) |
